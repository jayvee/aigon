## Why Aigon

- **Bring your own subscriptions** — use the AI products you already pay for (Claude, Codex, Gemini, Cursor); no separate Aigon billing, no token packs, no markup.
- **Stay in the flow when a subscription lapses** — switch agents or accounts mid-session instead of being stuck on one vendor's meter.
- **Spec-driven development you control** — features and research live as specs that can be built, shipped, reset, and rebuilt for clean handoffs and retries.
- **Autonomous and multi-agent workflows** — configure who drafts, implements, and reviews, then run long loops through a pre-built push/merge lifecycle.
- **Context that compounds** — specs and implementation summaries feed forward, so work accrues as durable project context instead of disappearing with a session.
- **Transparent, local, team-aligned** — agents run in plain tmux sessions you can attach to; it is CLI + git on your machine, with no required hosted platform, and fits GitHub PR review cleanly.

## Prerequisites

Before installing Aigon, make sure you have:

| Prerequisite | Required | Check | Why |
|---|---|---|---|
| **Node.js 18+** | Yes | `node -v` | Aigon is a Node.js CLI |
| **Git 2.20+** | Yes | `git --version` | Spec files are tracked in Git |
| **tmux** | For Fleet/worktree mode | `tmux -V` | Persistent agent sessions that survive terminal closes. Optional if you only use single-agent Drive mode on a branch. |

### Install prerequisites

**macOS (Homebrew):**

```bash
brew install node git tmux
```

**Ubuntu / Debian:**

```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo bash -
sudo apt-get install -y nodejs git tmux lsof
```

**Fedora:**

```bash
sudo dnf install nodejs npm git tmux
```

**Arch:**

```bash
sudo pacman -S nodejs npm git tmux
```

> **Note:** On Ubuntu/Debian, `nodejs` 18+ is required — the NodeSource script above installs the correct version.

## Installation

Install Aigon globally with npm:

**macOS:**

```bash
npm install -g @aigon/cli
```

**Linux:**

```bash
sudo npm install -g @aigon/cli
```

Then run the setup wizard:

```bash
aigon onboarding
```

The wizard walks you through prerequisites, terminal preference, agent CLI installation, an optional seed project, and starting the dashboard — all in one guided flow. It runs automatically the first time you use any `aigon` command, so you can also just skip straight to [Initialize a project](#initialize-a-project) and the wizard will launch on its own.

To re-run it at any time: `aigon onboarding --resume`

### Agent CLIs (manual install)

If you prefer to install agent CLIs yourself rather than through the wizard:

| Agent | Binary | macOS | Linux |
|-------|--------|-------|-------|
| Claude Code | `claude` | `brew install claude-code` | `sudo npm i -g @anthropic-ai/claude-code` |
| Gemini CLI | `gemini` | `npm i -g @google/gemini-cli` | `sudo npm i -g @google/gemini-cli` |
| Codex CLI | `codex` | `npm i -g @openai/codex` | `sudo npm i -g @openai/codex` |
| Cursor | `agent` | `curl https://cursor.com/install -fsS \| bash` | `curl https://cursor.com/install -fsS \| bash` |

> **Cursor:** The installer places `agent` in `~/.local/bin`. If the command isn't found after install, add it to your PATH: `echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc && source ~/.bashrc`. See the [Cursor CLI docs](https://cursor.com/docs/cli/installation) for more.

You don't need all of them — install whichever you have access to. Fleet mode works best with 2+ agents.

If you have been granted beta access to Aigon Pro, clone and link it after installing Aigon:

```bash
git clone https://github.com/jayvee/aigon-pro.git ~/src/aigon-pro
cd ~/src/aigon-pro && npm install
```

**macOS:**
```bash
npm link && cd ~/src/aigon && npm link @aigon/pro
```

**Linux:**
```bash
sudo npm link && cd ~/src/aigon && sudo npm link @aigon/pro
```

> **Access:** Aigon Pro is a private repository. If `git clone` fails with a permissions error, you have not yet been granted beta access. Contact [Sen Labs](https://senlabs.ai) to request it. Once granted, your GitHub account will be added as a collaborator and `gh auth login` will provide the necessary credentials.

### Optional tools

| Tool | What it enables | macOS | Linux |
|------|----------------|-------|-------|
| **GitHub CLI (`gh`)** | [GitHub Integration](/docs/guides/github-integration) — `feature-close` blocks on open PRs | `brew install gh` | See below |
| **gitleaks** | Secret scanning in pre-commit hooks and merge gates | `brew install gitleaks` | `sudo apt install gitleaks` (or see [releases](https://github.com/gitleaks/gitleaks/releases)) |
| **Semgrep** | SAST vulnerability scanning at merge gates | `brew install semgrep` | `sudo apt-get install -y python3-pip && pip3 install semgrep` |
| **Playwright** | Dashboard end-to-end tests (aigon contributors only) | `npx playwright install` (from the aigon repo) | `npx playwright install` then `sudo npx playwright install-deps` |

**Installing `gh` on Linux (Ubuntu / Debian):**

```bash
(type -p wget >/dev/null || (sudo apt update && sudo apt install wget -y)) \
  && sudo mkdir -p -m 755 /etc/apt/keyrings \
  && out=$(mktemp) && wget -nv -O$out https://cli.github.com/packages/githubcli-archive-keyring.gpg \
  && cat $out | sudo tee /etc/apt/keyrings/githubcli-archive-keyring.gpg > /dev/null \
  && sudo chmod go+r /etc/apt/keyrings/githubcli-archive-keyring.gpg \
  && sudo mkdir -p -m 755 /etc/apt/sources.list.d \
  && echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null \
  && sudo apt update \
  && sudo apt install gh -y
```

Then authenticate:

```bash
gh auth login
```

### Verify your setup

Run the doctor command to check that everything is configured correctly:

```bash
aigon doctor
```

Doctor checks Node.js version, Git availability, tmux, and which agent CLIs are in your PATH.

> **Git identity:** Aigon commits spec moves on your behalf. If you've never set up Git on this machine, do it now — otherwise your first feature start will fail:
> ```bash
> git config --global user.name "Your Name"
> git config --global user.email "you@example.com"
> ```

### Configure your terminal

Aigon launches agent sessions in terminal windows. First, create your global config:

```bash
aigon config init --global
```

#### macOS

Pick the terminal you use:

| Terminal | Command |
|----------|---------|
| **iTerm2** (recommended — agents run in tabs within a single window) | `aigon config set --global terminalApp iterm2` |
| **Warp** (agents run in side-by-side split panes) | `aigon config set --global terminalApp warp` |
| **Terminal.app** (macOS built-in) | `aigon config set --global terminalApp apple-terminal` |

> **Default:** Terminal.app. If you're on macOS and happy with Terminal.app, you can skip this step.

#### Linux

Aigon auto-detects your terminal on Linux — you can skip this step if you're happy with auto-detection. To set one explicitly:

| Terminal | Command |
|----------|---------|
| **GNOME Terminal** | `aigon config set --global linuxTerminal gnome-terminal` |
| **Kitty** | `aigon config set --global linuxTerminal kitty` |
| **xterm** | `aigon config set --global linuxTerminal xterm` |

> **Headless / server Linux:** No terminal config is needed. Agent sessions run inside tmux and you connect to them with `tmux attach -t <session-name>`.

## Initialize a project

**New to Aigon?** Start with the [Brewboard Tutorial](/docs/guides/brewboard-tutorial). It clones a seed project, walks through the full feature loop, and confirms your install is working — takes about 10 minutes. Come back here when you're ready to use Aigon on your own project.

**Have your own project ready?** Continue below.

---

Navigate to any Git repository and run:

```bash
cd /path/to/your/project
aigon init
```

This creates the `docs/specs/` folder structure with Kanban lanes for features, research, and feedback.

Then install the agents you want to use:

```bash
aigon install-agent cc gg          # Claude Code + Gemini CLI
```

| Code | Agent | Aigon command prefix |
|------|-------|----------------------|
| `cc` | Claude Code | `/aigon:` |
| `gg` | Gemini CLI | `/aigon:` |
| `cx` | Codex CLI | `$aigon-` (skill mention) |
| `cu` | Cursor | `/aigon-` |

Install only the agents you have access to. You can add more later with `aigon install-agent`.

> **Tip:** `install-agent` checks whether the agent CLI binary is in your PATH and warns you if it's missing, so you'll know immediately if you need to install something first.

## Three ways to interact

The dashboard is the recommended starting point — it gives you a visual overview of the pipeline and lets you start, monitor, and close features with a click. Agent slash commands and the CLI are there when you want more control or want to script things.

| | Dashboard | Agent Commands | CLI |
|---|---|---|---|
| **Where** | Browser UI | Inside an agent session | Terminal / shell |
| **Best for** | Visual overview, monitoring, starting features | Writing specs, implementing features | Orchestration, setup, scripting |
| **Example** | Click "Start" on a kanban card | `/aigon:feature-do 07` | `aigon feature-start 07 cc gg` |

All three surfaces drive the same underlying workflow. Mix them freely.

### Start the dashboard

Register your project and start the server:

```bash
aigon server add /path/to/your/project
aigon server start
```

Open [http://localhost:4100](http://localhost:4100). The Pipeline view shows your features across Kanban lanes; clicking **Start** on a card launches an agent session directly.

#### Persistent mode (survives reboots)

**macOS** — installs a launchd service, starts automatically:

```bash
aigon server start --persistent
```

**Linux with systemd (desktop or server with an active user session):**

```bash
aigon server start --persistent
```

> **Note:** On a headless Linux server, systemd user services require lingering to be enabled so the service survives logout: `sudo loginctl enable-linger $USER`. Then run `aigon server start --persistent`.

**Linux without systemd (containers, WSL, minimal installs)** — run in the background instead:

```bash
nohup aigon server start > ~/.aigon/logs/server.log 2>&1 &
```

## Your first feature

The fastest path from idea to implementation is `feature-now`. Describe what you want — the agent creates a spec, assigns an ID, sets up the workspace, and starts implementing:

```
/aigon:feature-now dark-mode
Add a dark mode toggle to the top-right of the site header.
Use CSS custom properties for theming. Default to system preference.
Store the user's choice in localStorage.
```

This is a normal agent conversation — you can go back and forth, refine requirements, ask questions, and iterate on the spec before the agent starts coding. The difference is that the conversation is anchored to a feature spec with acceptance criteria, so decisions are captured rather than lost in chat history.

### Step by step (the full loop)

For more control, use the individual commands:

```
# 1. Create a feature — describe what you want
/aigon:feature-create dark-mode
Add a dark/light mode toggle with system preference detection...

# 2. Prioritise — assigns an ID, moves to backlog
/aigon:feature-prioritise dark-mode
# → Assigned ID: 07

# 3. Start — creates the branch and moves the spec to in-progress
/aigon:feature-start 07

# 4. Do the feature — the agent reads the spec and implements it
/aigon:feature-do 07

# 5. Agent implements the spec...
#    → explores the codebase, writes code
#    → commits, writes implementation log
#    → signals "submitted" when done

# 6. Merge to main
/aigon:feature-close 07
```

Each step moves the feature through the pipeline: `inbox → backlog → in-progress → done`.

## Set up the local proxy (optional)

Aigon can route `.localhost` subdomains to its services, giving you clean URLs instead of `localhost:PORT`:

| Without proxy | With proxy |
|---|---|
| `http://localhost:4100` | `http://aigon.localhost` |
| `http://localhost:3001` | `http://brewboard-cc.localhost` |
| `http://localhost:3002` | `http://brewboard-gg.localhost` |

This is especially useful in Fleet mode, where each agent gets its own dev server on a different port. Instead of remembering which port belongs to which agent, you get named URLs that match the agent and project.

The proxy uses [RFC 6761](https://www.rfc-editor.org/rfc/rfc6761) `.localhost` domains, which resolve to `127.0.0.1` automatically — no DNS or `/etc/hosts` changes needed.

### Quick start (runs until you stop it)

```bash
aigon proxy start
```

### Persistent install (survives reboots, recommended)

This installs a lightweight system daemon on port 80:

```bash
aigon proxy install          # requires sudo — installs a LaunchDaemon (macOS) or systemd unit (Linux)
```

Verify it's running:

```bash
aigon proxy-setup
```

You should see `✅ aigon-proxy: running`. If port 80 is in use by another service (Apache, nginx), you'll see a warning — stop that service first or the proxy will fall back to a non-standard port.

> **Tip:** If you set up the proxy before running the [Brewboard Tutorial](/docs/guides/brewboard-tutorial), the dashboard and dev server previews will use the clean `.localhost` URLs automatically.

To remove the daemon later: `aigon proxy uninstall`.

## Updating Aigon

```bash
npm update -g @aigon/cli
```

Project templates and agent configs update automatically the next time you start an agent session. To force a sync immediately:

```bash
cd /path/to/your/project
aigon update
```

> If the dashboard is running, `aigon update` restarts it automatically to pick up the changes.
