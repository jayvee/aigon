Aigon is a spec-driven multi-agent harness — orchestrate Claude Code, Gemini CLI, and Codex CLI from one Kanban board, one CLI, or one slash command.

This page is a **projection of [the internal matrix](https://github.com/jayvee/aigon/blob/main/docs/competitive/matrix.md)** — five observable axes across the tools Aigon is most often compared against. For the full 10-axis grid, the per-tool deep dives, and the per-cell sources, see `docs/competitive/` in the repo.

## What "spec-driven multi-agent harness" means

Three orthogonal terms — and Aigon sits at the intersection of all three:

- **Harness** — everything around the model: scaffolding, context, tool dispatch, safety gates. *Cuts cleanly against IDE-embedded tools (Cursor) and platforms (Devin).*
- **Multi-agent** — coordinating multiple agents on the same goal. *Cuts cleanly against single-agent tools (Aider, OpenCode, Crush).*
- **Spec-driven** — Markdown specs as first-class executable artifacts. *Cuts cleanly against chat-first orchestrators (Cline Kanban) and session-first agents.*

## Trade-offs, axis by axis

For each axis, where is Aigon's choice genuinely a trade-off — not a unilateral win?

### Unit of work — Markdown spec
**Aigon picks it because** reasoning against an artifact beats reasoning against chat history; specs survive context resets and are diffable in PR review.
**It costs** upfront friction; Aigon is hostile to vibe-coding. If you want to start typing prompts, use Aider or OpenCode.

### Source of truth — Markdown specs in git
**Aigon picks it because** diffs, blames, PRs, and review tools all just work. Nothing is hosted; nothing is proprietary.
**It costs** no rich linking, no semantic search, no GUI comments on cards. Cline Kanban's hosted board state is more ergonomic for live review.

### Multi-agent posture — parallel competition + cross-agent eval
**Aigon picks it because** different models catch different issues; the best implementation wins on its own diff merits.
**It costs** N× the tokens on Fleet runs. Only worth it for high-stakes features. Solo Drive mode is the cheaper default.

### Model flexibility — BYO subscriptions
**Aigon picks it because** you use the tier you already pay for; no markup, no lock-in.
**It costs** setup per CLI; no curated default. OpenCode (75+ providers, mid-session switch) is more flexible at the single-agent level.

### Pricing — free OSS + BYO
**Aigon picks it because** the project doesn't sit in your billing path.
**It costs** zero cloud convenience; you handle keys, you handle compute. Devin's $20/mo + ACU model is friction-free for teams that prefer not to manage local infrastructure.

## Where Aigon falls short

The honest list. The full version lives in [`docs/competitive/weaknesses.md`](https://github.com/jayvee/aigon/blob/main/docs/competitive/weaknesses.md).

- **No IDE experience.** CLI + web dashboard only. Cursor, Cline, and Kiro all offer native IDE experiences.
- **No zero-config onboarding.** Cline Kanban (`npm i -g cline`) and one-click IDE tools are productive in minutes; Aigon takes longer to learn.
- **No cloud execution.** Everything runs locally in tmux. Devin, Jules, and Cursor Background Agents win for teams that want zero local infrastructure.
- **No first-class local-model support.** OpenCode (75+ providers, Ollama built-in) and Goose are better today; this is a tracked gap (`local-model-first-class-support`).
- **No per-agent / per-feature cost dashboard.** Roo Code surfaced cost inline; Aigon has telemetry but no spend view yet (`agent-cost-dashboard`).
- **Smaller community.** Cline Kanban (5M+ installs), OpenCode (132k stars), GSD (57.6k stars), Aider all have more users, more tutorials, more battle-testing.
- **No embedded visual testing.** Cursor's aggregated diff view and embedded browser are stronger for visual / QA-heavy workflows.

## Three categories — where Aigon sits

The space has fractured. The mental model below avoids comparing tools that don't share a shape:

1. **Single-agent coding tools** — Claude Code, Gemini CLI, Codex CLI, Aider, OpenCode, Crush. *Engines Aigon orchestrates, or one-session pair programmers.*
2. **IDE-embedded multi-agent** — Cursor 3 Agents Window. *Parallel inside the editor, single vendor on the model side.*
3. **Multi-agent orchestrators on real branches** — Aigon, Cline Kanban, Google Scion, GSD. *Worktrees, multiple agents, lifecycle management. Aigon's actual peer group.*

A fourth shape — **autonomous cloud agents** like Devin and Jules — is a different category entirely: cloud sandbox, vendor-locked model, hosted workspace, async execution.

## Complementary usage

Aigon works best **alongside** other tools. Its strength is orchestration — use specialised tools for implementation.

```bash
aigon feature-start 42 cc gg cu      # Claude, Gemini, and Cursor compete
aigon feature-open 42                # Each gets its own worktree + tmux session
aigon feature-eval 42                # Compare diffs and pick a winner
```

- **Aigon + Cursor** — Cursor as the editor / one of the racers; Aigon as the lifecycle and the merge gate.
- **Aigon + Aider** — Aider for exploratory prototyping; Aigon's spec lifecycle for the structured version.
- **Aigon + OpenCode** — OpenCode for breadth of providers (incl. local models); Aigon for orchestration and evaluation.
- **Aigon + Cline** — Cline for interactive single-agent IDE work; Aigon for the broader Kanban lifecycle.

## Key takeaway

Aigon is the only tool that is *all three* of:

- **A harness** (not a model, not an IDE, not a hosted product)
- **Multi-agent** (parallel competition + cross-agent evaluation, not just sequential delegation)
- **Spec-driven** (Markdown in git as the unit of work, not chat history or hosted board state)

If you want any one of those three, there are good single-purpose tools — pick from the list above. If you want them coordinated — that's Aigon.
