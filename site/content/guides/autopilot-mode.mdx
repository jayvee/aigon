Autonomous mode lets you start a feature and walk away. The **AutoConductor** — a dedicated background session — watches workflow state and triggers the next stage automatically. When it finishes (or stops), the feature falls back to normal manual workflow.

## How it works

The AutoConductor is a dedicated `auto` tmux session that polls the workflow-core engine every 30 seconds. When a stage completes, it invokes the next command automatically.

```
feature-autonomous-start
        ↓
AutoConductor spawns (auto tmux session)
        ↓
Agents implement in parallel
        ↓ all submitted
Solo mode → feature-close (done)
Fleet mode → feature-eval → exit, print next step (pick winner)
```

The AutoConductor is deliberately lightweight — it is not a lifecycle authority. It holds no orchestration state. If it dies, the feature continues normally and can be completed manually.

## Starting autonomously

### From the CLI

```bash
# Solo — one agent, fully hands-off through close
aigon feature-autonomous-start 42 cc

# Fleet — two agents, stop after eval (then pick winner manually)
aigon feature-autonomous-start 42 cc gg --eval-agent=gg --stop-after=eval

# Stop early — just implement, then hand back control
aigon feature-autonomous-start 42 cc --stop-after=implement
```

### From the dashboard

Click **Start Autonomously** on any backlog feature card. A modal lets you choose:

- **Implementation agents** — which agents implement in parallel
- **Evaluator agent** — which agent runs the evaluation (Fleet only)
- **Stop after** — where autonomy ends

## Modes

| Mode | Agents | What AutoConductor does |
|------|--------|------------------------|
| **Solo** | 1 | Waits for agent to submit → invokes `feature-close` → done |
| **Fleet** | 2+ | Waits for all agents to submit → invokes `feature-eval` → exits with next-step instructions |

Solo mode has no evaluation stage — the single agent is the winner by default and the feature closes automatically.

Fleet mode stops after evaluation in v1. Selecting the winner and closing is still a manual step:

```bash
aigon feature-close 42 cc       # after eval, pick the winner
```

## `--stop-after` options

| Value | Behaviour |
|-------|-----------|
| `close` | Default. Fully hands-off in solo mode. Fleet falls back to `eval` (no auto-close in v1) |
| `eval` | Stop after evaluation — return control before winner selection |
| `implement` | Stop after all agents submit — skip eval entirely |

## Monitoring

While the AutoConductor is running, the dashboard shows a **Running autonomously** badge on the feature card. When it finishes or is killed, the badge disappears.

To watch the AutoConductor log directly:

```bash
# Check whether the auto session is alive and the current engine state
aigon feature-autonomous-start status 42

# Attach to the AutoConductor output
tmux attach -t <repo>-f42-auto
```

## Resilience

If the AutoConductor session dies mid-run, implementation and eval sessions continue unaffected. Complete the feature manually:

```bash
aigon feature-eval 42             # if agents haven't been evaluated yet
aigon feature-close 42 cc         # merge the winner
```

No recovery state to clean up — the feature is just in whatever stage it was in when the AutoConductor stopped.

## When to use

Autonomous mode works best when:

- The spec has clear, machine-verifiable acceptance criteria
- You trust the agent to implement without mid-stream decisions
- You want parallel Fleet runs without babysitting

For features that need ongoing direction or spec refinement, stick with manual Drive mode and check in as the agent works.
