## Key Findings

### 1. Where in the feature lifecycle should PRs be created?

**Recommendation: Create draft PR at `feature-start`, convert to ready-for-review at `feature-submit`.**

Three options evaluated:

| Option | When | Pros | Cons |
|--------|------|------|------|
| A. At `feature-start` (draft) | After worktree + first commit | Early visibility, CI starts running, reviewers can watch progress | Draft PRs can be noisy; requires branch push immediately |
| B. At `feature-submit` | After agent signals done | Clean PR with final code; no draft noise | Late visibility; no CI feedback during development |
| C. Separate `feature-pr` command | Manual trigger | Maximum control | Extra step to remember; breaks automation flow |

**Option A is best** because it mirrors how Devin and GitHub Copilot Coding Agent work — both create draft PRs early. The worktree setup in `lib/worktree.js` (`setupWorktreeEnvironment()` at line ~1180) already commits a setup commit, so a branch exists to push. The draft PR gives teams visibility without noise.

At `feature-submit` (when `aigon agent-status submitted` fires), the draft converts to ready-for-review and reviewers are assigned. This maps to the existing shell trap signal infrastructure — the `trap EXIT` handler in `buildAgentCommand()` already fires `agent-status submitted` on exit 0 for implementation sessions.

### 2. What should the PR contain?

The PR description should be auto-generated from Aigon's existing artifacts:

**PR Title:** `feat({feature-id}): {spec title}` — derived from the spec filename/heading.

**PR Body template:**

```markdown
## Summary
{First paragraph from spec's ## Goal or ## Description section}

## Acceptance Criteria
{Parsed from spec — lib/validation.js already parses these}

## Agent Details
- **Agent:** {agent-id} ({agent display name})
- **Feature:** {feature-id}
- **Mode:** {Drive/Fleet}

## Implementation Log
{Link to or inline of the implementation log}

---
Generated by [Aigon](https://aigon.build) | Agent: {agent-id}
```

This follows the pattern established by Devin (agent identity block + linked artifacts) and Graphite's best-practice guide (summary, change type, linked issues, risk callouts).

**Agent-specific PR templates:** Support `.github/PULL_REQUEST_TEMPLATE/aigon_pr_template.md` as an override, following Devin's pattern of `devin_pr_template.md`. This lets teams maintain separate standards for agent PRs vs human PRs.

### 3. How does `feature-review` change when PRs exist?

Two modes, both valuable:

**Mode A — PR-integrated review (recommended for teams):**
- Review agent posts findings as **PR review comments** using `gh pr review --comment --body "..."` or line-level comments via `gh api`.
- Changes requested by the review agent become **PR change requests** (`gh pr review --request-changes`).
- Approval becomes `gh pr review --approve`.
- The existing `review-state.json` in `lib/feature-review-state.js` tracks local state; PR review status is the external source of truth for teams.

**Mode B — Local review with PR sync (hybrid):**
- Review agent writes to local log (current behavior).
- After review completes, a summary is posted to the PR as a comment.
- This preserves Aigon's offline-first design while adding PR visibility.

The config should let teams choose: `prReview: 'pr-comments' | 'local-with-sync' | 'local-only'`.

### 4. Should `feature-close` be blocked until PR is merged?

**Yes, when `requireApproval: true` is set in config.**

Currently, `feature-close` in `lib/feature-close.js` runs these phases: resolveCloseTarget → checkResumeState → autoCommitAndPush → mergeFeatureBranch → recordCloseTelemetry → closeEngineState → commitSpecMove → cleanupWorktreeAndBranch.

With PRs enabled, the flow changes:

- **If `requireApproval: false`:** `feature-close` merges the PR via `gh pr merge --squash --delete-branch --auto` and proceeds as normal. The `mergeFeatureBranch()` phase (line ~277) is replaced by PR merge.
- **If `requireApproval: true`:** `feature-close` checks PR approval status via `gh pr view --json reviewDecision`. If not approved, it blocks with a clear message: "PR #{number} requires approval before close. Run `aigon feature-close {id}` again after approval."

This maps to the workflow-core state machine — a new guard `prApproved` could gate the `closing` → `done` transition when PR workflow is enabled.

### 5. How does the worktree → branch → PR flow work?

Current flow in `lib/worktree.js` and `lib/commands/feature.js`:

1. `feature-start` creates worktree with `git worktree add {path} -b {branch}` (branch: `feature-{id}-{agent}-{desc}` for Fleet, `feature-{id}-{desc}` for Drive)
2. Agent works in worktree, commits locally
3. `feature-submit` — agent signals done
4. `feature-close` — merges branch to main locally, removes worktree

**With PRs, the new flow:**

1. `feature-start` creates worktree + branch (unchanged)
2. **NEW:** Push branch to remote: `git push -u origin {branch}`
3. **NEW:** Create draft PR: `gh pr create --draft --title "..." --body "..." --base main`
4. Store PR URL in `.aigon/workflows/features/{id}/snapshot.json` (new field: `prUrl`, `prNumber`)
5. Agent works in worktree, commits + pushes periodically
6. `feature-submit` — convert draft to ready: `gh pr ready {number}`; assign reviewers: `gh pr edit {number} --add-reviewer {reviewers}`
7. `feature-review` — post review as PR comments
8. `feature-close` — merge via `gh pr merge {number} --squash --delete-branch` instead of local merge

The branch naming convention already works well for PRs — `feature-42-cc-user-auth` is human-readable in GitHub's PR list.

### 6. What config shape makes sense?

```jsonc
// .aigon/config.json (project-level)
{
  "prWorkflow": {
    "enabled": false,           // Master toggle
    "host": "github",           // "github" | "gitlab" (future)
    "baseBranch": "main",       // Target branch for PRs
    "draft": true,              // Create as draft initially
    "reviewers": [],            // Default reviewers: ["@team-leads", "alice"]
    "labels": ["aigon"],        // Labels to apply
    "requireApproval": false,   // Block feature-close until PR approved
    "mergeStrategy": "squash",  // "squash" | "merge" | "rebase"
    "deleteBranch": true,       // Delete branch after merge
    "autoMerge": false,         // Enable gh auto-merge
    "template": null            // Path to PR template, or null for auto-generated
  }
}
```

This sits alongside the existing `workflow` and `security` config blocks. The `prWorkflow` key is distinct and self-documenting. All fields have sensible defaults so `"prWorkflow": { "enabled": true }` is sufficient to get started.

**Config access:** Via existing `loadProjectConfig()` / `getEffectiveConfig()` in `lib/config.js`. No new config infrastructure needed.

### 7. How do we handle the `gh` CLI dependency?

**`gh` should be required when `prWorkflow.enabled: true`, not globally.**

Detection: Check `which gh` and `gh auth status` at `feature-start` time. If `gh` is not installed or not authenticated, fail with a clear error:

```
Error: PR workflow is enabled but `gh` CLI is not installed.
Install it: brew install gh && gh auth login
Or disable PR workflow: aigon config set prWorkflow.enabled false
```

`aigon doctor` should include a PR workflow check when it's enabled: verify `gh` exists, is authenticated, and has the correct scopes.

**Multi-host support (future):** Abstract behind a `PRProvider` interface:

```javascript
// lib/pr-provider.js (future)
const providers = {
  github: { create, merge, review, status, ... },  // uses gh
  gitlab: { create, merge, review, status, ... },   // uses glab
};
```

For v1, hardcode GitHub via `gh`. The `host` config field reserves the extension point. GitLab support via `glab` would be low effort — the flag mapping is ~90% identical (e.g., `--body` → `--description`, `--base` → `--target-branch`).

Bitbucket has no official CLI and a fragmented ecosystem (bb-cli, bbt, bkt, or raw REST API). Recommend deferring Bitbucket to v2 or later.

### 8. What happens in Fleet mode with multiple agents?

**Each agent gets its own PR.** This matches how Fleet already works — each agent gets its own branch (`feature-{id}-{agent}-{desc}`).

Benefits of per-agent PRs:
- CI runs independently per implementation
- Reviewers can compare PRs side-by-side
- `feature-eval` can link to all PRs in the evaluation doc
- The winning agent's PR gets merged; losing PRs are closed

**Flow in Fleet mode:**
1. `feature-start cc gg cx` — 3 worktrees, 3 branches, 3 draft PRs
2. Each agent pushes to its branch independently
3. `feature-submit` per agent → each draft converts to ready
4. `feature-eval` — evaluator reviews all 3 PRs, writes evaluation
5. `feature-close {winner}` — winner's PR merged, others closed via `gh pr close {number}`
6. `cleanupLosingBranches()` (line ~773 in feature-close.js) already handles branch deletion — extend it to also close PRs

### 9. How do automated agent reviews (CodeRabbit, Vercel Agent) fit in?

All major automated reviewers (CodeRabbit, Vercel Agent, Copilot Review) trigger via **webhooks on PR creation/update** — no Aigon integration needed. Once Aigon creates a PR, these tools activate automatically if installed on the repo.

**Should Aigon wait for their results?** Optional, via config:

```jsonc
{
  "prWorkflow": {
    "waitForChecks": false,      // Block feature-close until all PR checks pass
    "requiredChecks": []         // Specific check names to wait for, e.g. ["coderabbit"]
  }
}
```

Implementation: Poll `gh pr checks {number} --json name,state` periodically. Or use `gh pr merge --auto` which natively waits for required status checks.

**Recommendation:** Don't build custom polling. Use `gh pr merge --auto` when `autoMerge: true` is set — GitHub handles the waiting natively.

### 11. How does this interact with `feature-eval`?

`feature-eval` (Fleet mode comparison) should:

1. **Link to PRs** in the evaluation document — each agent's section includes its PR URL
2. **Use PR diffs** as input — `gh pr diff {number}` provides a clean diff, potentially better than raw `git diff` across branches
3. **Post eval results** as comments on the winning PR (optional, via config)
4. **Close losing PRs** as part of the eval → close flow

The eval document already lives at `evaluations/feature-{id}-eval.md`. Adding PR links is a template change, not a code change.

### 12. What's the UX for the dashboard?

The dashboard (served by `lib/dashboard-server.js`, status collected by `lib/dashboard-status-collector.js`) should show:

- **PR badge** on feature cards — clickable link to the PR on GitHub
- **PR status** — draft, open, approved, merged, closed
- **Check status** — CI passing/failing (from `gh pr checks`)
- **Review status** — pending, approved, changes requested

Implementation: Add `prUrl`, `prNumber`, `prStatus` to the feature status payload in `lib/dashboard-status-collector.js` (`collectFeatureDeepStatus()`). The dashboard frontend renders these from the existing `validActions` API response pattern.

This is a natural fit for **Aigon Pro** — the dashboard PR widget adds value primarily for teams. Free tier could show a simple PR link; Pro could add rich status, check results, and review timeline.

### OSS vs Pro Boundary

**In OSS (`aigon`):**
- Config schema (`prWorkflow` in `.aigon/config.json`)
- `gh` CLI detection and `aigon doctor` check
- Branch push in `lib/worktree.js` (needed regardless of PRs for remote backup)
- Basic `lib/pro-bridge.js` extension points for PR lifecycle hooks
- PR URL storage in workflow snapshot

**In `@aigon/pro`:**
- PR creation with auto-generated description (spec parsing, log inclusion)
- PR description template system (`.github/PULL_REQUEST_TEMPLATE/aigon_pr_template.md`)
- Reviewer assignment logic
- PR status polling and dashboard widgets
- `feature-close` PR merge integration
- `feature-eval` PR linking and losing-PR closure
- Automated review integration (waitForChecks, requiredChecks)

This follows the existing pattern: OSS provides the plumbing (branch push, config, bridge hooks), Pro provides the workflow intelligence (PR creation, description generation, status tracking).

### Competitor Positioning Analysis

| Tool | PR Creation | Team Tier Price | PR Gated Behind Paid? |
|------|------------|----------------|----------------------|
| GitHub Copilot Agent | Draft PRs from issues | $19-39/user/mo | Yes (agent itself is gated) |
| Cursor | No native PR creation | $40/user/mo | N/A (Bugbot add-on for review) |
| Windsurf | No native — uses `gh` in workflows | $30/user/mo | N/A |
| Devin | PR is primary deliverable | $500/mo (Team) | No — all tiers create PRs |

**Aigon positioning:** Gating PR workflow behind Pro is defensible because:
1. Solo developers don't need PRs — Aigon's local review loop is sufficient
2. PRs are team infrastructure — they have no value for solo use
3. Competitors either gate the entire agent (Copilot) or charge high team prices (Devin $500/mo)
4. The free → Pro upsell path is natural: solo user joins team → team requires PRs → needs Pro

The elevator pitch: *"Your agents already write the code — now your team can review it the way they review everything else: pull requests."*

## Recommendation

**Implement PR workflow as a Pro feature using a phased approach:**

**Phase 1 — Foundation (OSS):**
- Add `prWorkflow` config schema with `enabled: false` default
- Add `gh` detection to `aigon doctor`
- Add branch push to `setupWorktreeEnvironment()` (useful even without PRs)
- Add `prUrl`/`prNumber` fields to workflow snapshot schema
- Add extension points in `lib/pro-bridge.js` for PR lifecycle hooks

**Phase 2 — Core PR Workflow (Pro):**
- Draft PR creation at `feature-start`
- Auto-generated PR description from spec + acceptance criteria
- Draft → ready conversion at `feature-submit`
- Reviewer assignment from config
- PR merge at `feature-close` (replacing local merge)
- Dashboard PR badge and status

**Phase 3 — Advanced (Pro):**
- `requireApproval` gating on `feature-close`
- PR-integrated review comments (review agent posts to PR)
- Fleet mode: per-agent PRs, losing PR closure
- `feature-eval` PR linking
- `waitForChecks` integration with `gh pr merge --auto`
- GitLab support via `glab` flag mapping

**Phase 4 — Future:**
- Bitbucket support
- Custom PR description templates (`.github/PULL_REQUEST_TEMPLATE/aigon_pr_template.md`)
- PR analytics in Aigon Pro insights
