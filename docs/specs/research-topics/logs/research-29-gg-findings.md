# Research Findings: PR Option for Team Workflows

**Agent:** gg
**Date:** 2026-04-12

## Findings

### Lifecycle & Flow
- **PR Creation:** The most natural point to create a PR is during `feature-submit`. When an agent finishes implementation and the tests pass, pushing the branch and opening a PR signals that the work is ready for human or automated review.
- **Worktree → Branch → PR Flow:** Aigon already creates isolated local branches for worktrees. The flow should be:
  1. Agent finishes work in worktree.
  2. `aigon feature-submit` is called.
  3. Aigon pushes the local branch to the remote (e.g., `git push origin feature-123-desc`).
  4. Aigon Pro (if enabled) intercepts the post-submit hook to create the PR.
- **`feature-close` Interaction:** If `usePRs` is true, `feature-close` should check the PR status. It should block closing if the PR is not merged (or not approved, based on config). Alternatively, `feature-close` could use the git host API to actually merge the PR, then perform its normal local cleanup (deleting the branch/worktree).
- **Fleet Mode:** Creating a PR for every agent in a Fleet run would be extremely noisy. For Fleet mode, PR creation should be deferred until a winner is selected during the evaluation phase, or a single "Draft PR" could be created that gets updated with the winning branch.

### PR Content & Review
- **Description Generation:** The PR body should be auto-generated by summarizing the `docs/specs/features/ID-name.md` file, the acceptance criteria, and the agent's final output log. This mimics features in Cursor Teams and Copilot Enterprise.
- **`feature-review` Changes:** When `usePRs` is enabled, the `feature-review` agent should still run locally to analyze the code, but instead of just writing to the local log, Aigon Pro can post the review findings as inline comments on the PR using the git host API.
- **Automated Agent Reviews:** Third-party tools like CodeRabbit or Vercel Agent will naturally trigger when the PR is opened. Aigon can optionally poll for these checks to pass before allowing `feature-close`.

### Configuration & Tooling
- **Config Shape:**
  ```json
  {
    "team": {
      "usePRs": true,
      "gitHost": "github",
      "requireApproval": true,
      "prTemplate": ".github/PULL_REQUEST_TEMPLATE.md",
      "defaultReviewers": ["@my-team"]
    }
  }
  ```
- **`gh` CLI vs API:** Relying on the `gh` CLI is a fast way to MVP the feature and handles local auth nicely (`gh pr create`). However, for a polished Pro feature, using direct API clients (like `@octokit/rest`) provides better error handling and doesn't require users to install additional CLI tools.

### Aigon Pro Positioning
- **Why it's Pro:** Solo developers get immense value from Aigon's local worktree loop. The moment you introduce a PR, you are dealing with team governance, compliance, and human collaboration. This is the definition of enterprise/team tier value.
- **Elevator Pitch:** "Your Aigon agents write the code locally. Aigon Pro seamlessly integrates them into your team's existing review culture by automatically drafting rich, context-aware pull requests."
- **Implementation Boundary:**
  - **OSS (`aigon`):** Adds `git push` capabilities to `feature-submit` and ensures appropriate lifecycle hooks (e.g., `runPostHook('feature-submit')`) are emitted.
  - **Pro (`@aigon/pro`):** Subscribes to the hooks via `pro-bridge.js`, handles PR description generation (using an LLM), interacts with the GitHub/GitLab API to create the PR, assigns reviewers, and adds PR status widgets to the dashboard.

## Recommendation

Implement PR integration as a flagship feature for **Aigon Pro**.
Modify the OSS `feature-submit` command to push the feature branch to the remote if a new `pushOnSubmit` config is enabled.
Use the existing `pro-bridge.js` to allow `@aigon/pro` to listen for a `feature-submitted` event. When triggered, the Pro module will generate a PR description and use the GitHub API (via a bundled octokit client, avoiding `gh` CLI dependencies) to open the PR.
Update `feature-close` in the OSS version to accept a `--remote-merge` flag, which the Pro module can utilize to ensure the PR is merged remotely before cleaning up local branches.
