# Feature: E2E Aigon Test Suite

## Summary

Create a comprehensive end-to-end test suite for Aigon backed by a deterministic fixture repo. Unlike the existing unit tests (which test individual functions in isolation), this suite spawns `aigon` as a subprocess against a real git repo seeded with known state, then asserts on stdout/stderr, exit codes, file system changes, and state transitions. Every major command path — feature lifecycle, research lifecycle, feedback lifecycle, board display, and config — is covered. Tmux commands are tested via a mock shim so CI doesn't need an actual terminal.

## User Stories

- [ ] As a developer modifying `lib/utils.js`, I can run `npm run test:e2e` and immediately know whether any CLI command broke — without manually opening `when-swell` and clicking through workflows
- [ ] As a reviewer evaluating a PR, I can see e2e test results in the output and trust that all kanban transitions were exercised
- [ ] As a contributor adding a new command, I have a clear pattern to follow for writing an e2e test against the fixture repo
- [ ] As a maintainer, I can reset the fixture repo to a known baseline with a single command and re-run any failing scenario in isolation

## Acceptance Criteria

### Fixture repo

- [ ] A fixture git repo exists at `test/fixture/` (committed to the Aigon repo, or generated by `test/setup-fixture.js`)
- [ ] The fixture can be recreated from scratch with `node test/setup-fixture.js` — no manual steps
- [ ] The fixture contains pre-seeded state: at least 2 features in each stage (inbox, backlog, in-progress, done), 2 research topics (inbox, in-progress, done), and 3 feedback items (inbox, triaged, actionable)
- [ ] Each test run copies the fixture to a temp dir (so tests are isolated and the fixture stays clean)

### Feature lifecycle paths

- [ ] `feature-create <name>` → file appears in `01-inbox/`, correct frontmatter
- [ ] `feature-prioritise <name>` → file moves to `02-backlog/`, ID assigned, board mapping updated
- [ ] `feature-setup <ID>` (Drive branch mode) → branch created, log file scaffolded
- [ ] `feature-setup <ID> cc gg` (Fleet mode) → two worktrees created, agent log files scaffolded
- [ ] `feature-do <ID>` (inside agent session) → prints instructions, detects Drive vs Fleet mode
- [ ] `feature-submit` → log status set to `submitted`, commit made
- [ ] `feature-eval <ID>` → evaluation file created in `logs/`
- [ ] `feature-close <ID>` → branch merged, spec moved to `05-done/`
- [ ] `feature-cleanup <ID>` → worktrees removed, branches deleted

### Research lifecycle paths

- [ ] `research-create <name>` → file in `01-inbox/`
- [ ] `research-prioritise <name>` → file in `02-backlog/`, ID assigned
- [ ] `research-setup <ID>` (Drive mode, no agents) → findings file scaffolded
- [ ] `research-setup <ID> cc gg` (Fleet mode) → two worktrees, two findings files
- [ ] `research-submit <ID> cc` → findings status set to `submitted`
- [ ] `research-synthesize <ID>` → synthesis file created
- [ ] `research-close <ID>` → spec moved to `04-done/`

### Feedback lifecycle paths

- [ ] `feedback-create <title>` → file in `01-inbox/`, ID assigned, frontmatter valid
- [ ] `feedback-list` → output includes all items with correct status labels
- [ ] `feedback-list --status=triaged` → filters correctly
- [ ] `feedback-triage <ID> --apply --yes` with `action=keep` → status updated to `triaged`
- [ ] `feedback-triage <ID> --apply --yes` with `action=mark-duplicate` → `duplicate_of` field set
- [ ] `feedback-triage <ID> --apply --yes` with `action=promote-to-feature` → new feature spec created in inbox

### Board command

- [ ] `aigon board` → exits 0, output contains column headers, feature IDs visible
- [ ] `aigon board --list` → exits 0, all stages listed
- [ ] `aigon board --features` → research section absent
- [ ] `aigon board --research` → features section absent
- [ ] `aigon board --active` → only in-progress items shown

### Config and setup

- [ ] `aigon init` → creates full `docs/specs/` directory structure with `.gitkeep` files
- [ ] `aigon profile detect` → returns a profile string (web/api/library/generic/etc.)
- [ ] `aigon profile set ios` → updates `.aigon/config.json`
- [ ] `aigon doctor` → exits 0 (or 1 with expected warnings), no crash

### Tmux / session commands (mocked)

- [ ] A `tmux` mock shim is on `PATH` during e2e tests; it records calls and returns configurable exit codes
- [ ] `feature-setup <ID> cc` → mock tmux session created with correct session name pattern `aigon-<ID>-cc`
- [ ] `sessions-close <ID>` → mock tmux kill-session called for all sessions matching feature ID
- [ ] `worktree-open <ID>` → mock tmux new-window or new-session called

### Regression guard

- [ ] After the CLI modularization (feature 68), all e2e tests still pass — confirming extracted modules are wired correctly through `shared.js`
- [ ] A dedicated smoke test runs `aigon help` and asserts exit code 0 and non-empty output — catches catastrophic require() failures instantly

### Test runner integration

- [ ] `npm run test:e2e` runs the full suite (separate from `npm test` unit tests)
- [ ] `npm run test:all` runs both unit and e2e suites
- [ ] Individual scenario groups can be run with `npm run test:e2e -- --grep feedback`
- [ ] Each test prints a clear pass/fail with the command that was run and actual vs expected output on failure

## Validation

```bash
npm run test:e2e
```

## Technical Approach

### Fixture structure

`test/fixture/` is a minimal git repo (with `.git/` committed via `git bundle` or generated by `setup-fixture.js`). The setup script:
1. `git init` in a temp dir
2. Creates the full `docs/specs/` tree
3. Seeds spec files with known IDs and frontmatter
4. Commits everything as a single "fixture baseline" commit

Each test copies this dir to `os.tmpdir()/aigon-e2e-<random>/` and runs commands with `cwd` set there.

### CLI invocation pattern

```js
function runAigon(args, { cwd, env = {} } = {}) {
    const result = spawnSync(process.execPath, [CLI_PATH, ...args], {
        cwd,
        env: { ...process.env, HOME: tempHome, PATH: mockBinDir + ':' + process.env.PATH, ...env },
        encoding: 'utf8'
    });
    return { stdout: result.stdout, stderr: result.stderr, exitCode: result.status };
}
```

All commands run with an isolated `HOME` (no pollution from the developer's `~/.aigon/`) and a `PATH` that puts the mock tmux shim first.

### Tmux mock shim

A tiny shell script at `test/mock-bin/tmux` that:
- Appends each invocation (args + cwd) to `$AIGON_MOCK_LOG` (a temp file)
- Exits 0 by default, or the code in `$AIGON_MOCK_EXIT` if set
- Allows tests to assert `tmux new-session -s aigon-7-cc` was called without actually spawning anything

### Assertion helpers

```js
function assertExitCode(result, code) { ... }
function assertStdoutContains(result, str) { ... }
function assertFileExists(cwd, relativePath) { ... }
function assertFileMoved(cwd, from, to) { ... }
function assertFrontmatterField(cwd, file, key, value) { ... }
```

### Isolation

- Each top-level test group gets a fresh copy of the fixture (not shared between groups)
- The mock tmux log file is per-test-run
- `process.cwd()` is never mutated — `cwd` is always passed explicitly to `spawnSync`

## Dependencies

- Feature 68 (CLI modularization) — done, this test suite validates it retroactively
- No external services required; tmux is mocked

## Out of Scope

- Testing the Radar daemon or dashboard HTML (complex subprocess lifecycle)
- Testing `dev-server start` (requires a real port and process)
- Testing autonomous loop (`feature-do --autonomous`) end-to-end — that spawns an agent CLI; a dry-run smoke test is sufficient
- Performance benchmarks

## Open Questions

- Should `test/fixture/` be committed as a git bundle, or always generated by `setup-fixture.js`? (Generated is more maintainable but adds ~200ms per test run for setup)
- Should e2e tests run in CI on every push, or only on release branches? (Full suite may take 30–60 seconds)
- Can we reuse the existing `withTempDir` / `withCapturedConsole` helpers from `aigon-cli.test.js`, or should e2e have its own runner file?

## Related

- Feature 68: CLI modularization (the direct motivation for this suite)
- `aigon-cli.test.js`: existing unit test file — e2e lives alongside it as `aigon-cli.e2e.test.js` or `test/e2e.test.js`
- `when-swell`: current manual test project — this suite makes manual testing optional for common paths
