v4 · F650 card hierarchy · Rules: dashboard-card-design.md · One dominant headline, quiet progress timeline, single primary action
1 · The Problem — Redundancy

Every card currently renders 3 separate zones that repeat the same 2 facts. State (e.g. "Implemented") and agent ("CC") each appear twice. Action buttons are a third row when they could sit inline.

2 · Vocabulary — precise verbs only
OldNewWhy
COMPLETEImplementedImplemented = agent wrote and pushed code. "Complete" says nothing about what was done.
REVISION DONERevisedAgent addressed review feedback. Revised is the specific act.
REVIEW DONECode reviewedA reviewer read and annotated the code. The next actor is the implementing agent, not the user.
SPEC REVIEW DONESpec reviewedA reviewer read the spec. The next actor is the spec owner agent, not the user.
RESEARCH DONEResearch readyFindings exist and are ready to evaluate. "Done" doesn't say what to do next.
Done (terminal)Closedaigon feature-close ran. Worktree gone. "Closed" is the precise terminal event.
awaiting review→ needs code reviewImperative. "Awaiting" is passive and vague about what kind of review.
Finished (unconfirmed)UnconfirmedSession ended without a signal. State is unknown — human must verify.
All completeAll implementedEach agent finished their implementation independently.
Spec driftState driftDrift is between engine snapshot and folder — not just in the spec.
Close recoveryRollbackSystem is undoing a partial close (git cleanup etc). The user just waits — this names the action.
3 · Card anatomy
Structure (max 3 rows per card)
[ID] [Feature name] ▾
[icon] [State label] · [Agent] · [Duration]
>_[CTA]···
[detail — only when it adds new info]
▾ expand toggle (top-right of every card) — opens the detail drawer: spec summary, full activity log, dependency graph, session transcript links.
Left border colour = urgency at a glance · Green = running · Teal = handoff · Amber = waiting · Red = error · Grey = static
Liveness dot (in agent token) · Green = alive · Amber = stale (>5m silent) · Red = dead · no dot = session ended
>_ button = open the agent's tmux session · only shown when an active session exists
Detail line = italic line below status · only when it adds genuinely new info (next-action hint, agent question, dep list, error context)
Multi-agent chips (sequential pipeline) · done agent muted · active agent has liveness dot · duration belongs to active agent

J — Feature sets · lightweight wrapper around full feature cards
A feature set is a teal-bordered container grouping related features. The set header shows aggregate state counts and a progress bar. When expanded, each member feature renders as a full feature card — identical to any standalone card — nested inside the wrapper. No compact rows, no special child layout: the same card design throughout.
