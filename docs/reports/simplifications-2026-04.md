### 3. Split `lib/dashboard-routes.js` into per-domain handler modules

**Inbox feature:** `simplify-dashboard-routes-2026-04` (filed via `feature-create`)

**Where:** `lib/dashboard-routes.js:1ŌĆō2079` (40+ route definitions in one file).

**Today:** A single `createDashboardRouteDispatcher()` returns an array of route handler objects, each with inline business logic, ad-hoc JSON parsing, and scattered error handling. The file imports `loadGlobalConfig` and many other modules just to satisfy a handful of routes; touching one route forces re-reading the whole file to confirm no shared state was missed.

**Proposed:** Create `lib/dashboard-routes/` with one file per domain (`recommendations.js`, `entities.js`, `set.js`, `config.js`, `analytics.js`, etc.). Each module exports a small array of route definitions. `dashboard-routes.js` becomes a 60-line aggregator that imports and concatenates them. Keep the existing route-definition shape ŌĆö this is purely a file-system reorganisation, not a re-architecture.

**Why now:** The dashboard is now the daily command centre (per memory), and route additions are increasingly common. Today's monolith makes route review and reasoning expensive. A per-domain split also makes it obvious which read-only routes can be safely served from cached snapshots later.
