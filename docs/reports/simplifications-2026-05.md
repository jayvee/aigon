### 1. Centralize duplicate fs/string utility primitives into `lib/fs-utils.js`

**Inbox feature:** `simplify-duplicate-utilities-2026-05` (filed via `feature-create`)

**Where:**
- `safeReadDir` defined in 4 files: `lib/dashboard-spec-index.js:27`, `lib/dashboard-status-collector.js:95`, `lib/feature-spec-resolver.js:67`, `lib/commands/feature.js:153`
- `stripAnsi` defined in 4 files: `lib/supervisor.js:214`, `lib/dashboard-server.js:268`, `lib/budget-poller.js:252`, `lib/quota-mid-run-detector.js:70`
- `safeStatMtimeMs` defined in 2 files: `lib/dashboard-status-collector.js:105`, `lib/dashboard-server.js:62`
- `parseSimpleFrontMatter` at `lib/dashboard-server.js:278` is a weaker reimplementation of `parseFrontMatter` in `lib/cli-parse.js`

**Today:** Ten function definitions are copy-pasted across the codebase. The copies are not identical ‚Äî `dashboard-status-collector.js`'s `safeReadDir` adds an `fs.existsSync` pre-check and an optional predicate filter that the other three copies lack. This divergence means a future caller cannot know which copy is "right" without diffing all four. `lib/utils.js` exists as a re-export hub but its charter is < 200 LOC and it has a circular-require constraint that prevents adding these helpers directly.

**Proposed:** Create `lib/fs-utils.js` (no circular-require risk) exporting the authoritative `safeReadDir(dir, predicate = null)`, `safeStatMtimeMs(filePath)`, and `stripAnsi(str)`. Replace all local copies with imports. Also replace `parseSimpleFrontMatter` in `dashboard-server.js` with a direct import of `parseFrontMatter` from `lib/cli-parse.js` (it's already imported lazily there anyway). Zero behaviour change ‚Äî just consolidation.

**Why now:** The `dashboard-status-collector.js` copy already silently diverged from the others; the next change to any of these helpers will hit at least four files unless there's a canonical home. AI agents reading any of the four files see a locally-defined copy with no signal that a canonical version exists elsewhere.

### 3. Wire `readComplexityFromSpec` through `specIndex` to eliminate per-feature disk reads

**Inbox feature:** `simplify-complexity-spec-index-2026-05` (filed via `feature-create`)

**Where:** `lib/dashboard-status-collector.js:85‚Äì92` (`readComplexityFromSpec`), called at lines 809, 902, and 1251; `lib/spec-recommendation.js:24‚Äì50` (`readSpecRecommendation` ‚Äî does the actual `readFileSync`).

**Today:** `readComplexityFromSpec(specPath)` calls `readSpecRecommendation(specPath)` which opens the spec file, reads it with `fs.readFileSync`, and runs `parseFrontMatter` ‚Äî all to extract the `complexity:` frontmatter field. But `collectFeatures` already builds a `specIndex` (via `dashboardSpecIndex.getRepoSpecIndex`, line 733) that stores `entry.frontmatterRaw` for every spec. For a repo with 20 active features + 10 inbox + 10 backlog, that's 40 extra `readFileSync` calls per dashboard refresh that serve data already in memory.

**Proposed:** Add an optional `specIndex` parameter to `readComplexityFromSpec(specPath, specIndex = null)`. When `specIndex` is provided, look up `specIndex.materialized.byPath.get(specPath)?.frontmatterRaw?.complexity` before falling back to the existing disk read. Pass the already-built `specIndex` through from `collectFeatures` and `collectResearch`. The fallback ensures nothing breaks if the index is stale or missing.

**Why now:** The dashboard is the daily command centre (always-on per project guidelines). Per-feature disk reads in the refresh hot path add latency that scales with repo size. This is the smallest targeted fix available for dashboard refresh performance ‚Äî the spec index is already built and the lookup is a single Map `.get()`.

**Scores:** maintainability 3 / readability 2 / AI-clarity 2 / extensibility 1 / performance 5 ‚Äî **total: 13 / 25**

**Effort:** small
