Design proposal · for review before spec-ing

Decluttering the Aigon pipeline: cards that answer one question

Four months of accreted capability has turned feature and research cards into control panels. This proposal separates glancing from inspecting from acting, reshapes the board around where attention is needed, and backs it with a real design-pattern reference. Five layout concepts follow — pick a direction and I'll turn it into an aigon feature set.

Basis: live board + e2e mock sandbox (fleet, solo, autonomous, sets, frozen mock agents) · docs/card-design-wireframe.html (~60 documented card states) · pipeline.js / card-presentation / kanban.css source review

Part 1

Diagnosis — why cards stopped being readable

A single in-progress card can currently stack seventeen distinct section types: id, title, four badge kinds (drift, escalation, lease, scheduled), spec author, headline, timeline, agent summary, dependency chain, autonomous controller panel, autonomous plan, idle badge, startup phase, nudge chips, per-agent sections (each with its own status, terminal button, peek eye, dev-server link, overflow menu), evaluation section, reviewer panels, spec-review block, review-cycle history, ready-to-close banner, GitHub section, close-failure panel, and the card action row. The F650 presentation model tamed the severity chaos (one dominant headline), but the quantity chaos remains: everything still renders inline, on a card that is often 220px wide.

The five problems, named

  • P1 — Redundant narration. Headline, controller panel, plan section, and agent rows all describe the same fact. On F650's own terms there is one dominant state — but three subordinate sections re-state it.
  • P2 — No altitude separation. The card is simultaneously a glanceable tile, a history log, and a control panel. Meanwhile a full detail surface already exists (the drawer: Spec / Status / Events / Agents / Stats / Code Changes / Agent Log / Control) — the card duplicates it instead of pointing at it.
  • P3 — Uniform columns, non-uniform attention. grid-template-columns: repeat(5, minmax(220px, 1fr)) gives Inbox and an overflowing In-Progress identical width. At 1680px the Closed column literally clips off-screen while Evaluation sits empty.
  • P4 — Set chrome tax. Group-by-set repeats a ~150px header block (3 buttons + 2 status rows + progress bar) per set per column — the chrome outweighs the cards it groups.
  • P5 — No enforced vocabulary. Cards are built by string concatenation in pipeline.js (2,400 lines) with a parallel variant in monitor.js. Every new capability invents its own chip/panel/button style; nothing structurally prevents clutter from re-accreting.

Part 2

The design model — glance, focus, act

Every concept below applies the same rule. A pipeline card exists to answer exactly four questions in under a second:

QuestionAnswered byBudget
What is this?ID + title + set/lease badges2 lines
What's happening now?One headline (state verb + agent + age)1 line
Does it need me?Severity color of that one headline0 extra
What's my one next move?Exactly one primary action1 button

Everything else — per-agent sessions, timelines, review history, GitHub checks, autonomous stage tracks, close blockers — is focus-level information. It lives one gesture away (expand, inspector, or drawer), never on the resting card. The server already computes almost all of this (cardPresentation, closeReadiness); this is a presentation-layer re-layering, not a data-model rewrite.

Part 3 · Layout concepts

Five concepts, composable — A+B+E is the recommended core

A

Attention-weighted board

recommended coreanswers "widen the in-progress row"low risk

Columns stop being equal. In-Progress and Evaluation get 2–3× width (room for full agent detail); Inbox, Backlog, and Closed collapse into narrow rails — a count, plus one-line micro-cards. Empty columns collapse to a labelled sliver. A rail expands to a full column on click (pinned per repo in localStorage). The board finally fits without horizontal clipping because width goes where the work is.

B

Two-state cards — collapsed by default, expand for detail

recommended coreyour expand/contract ideaworks in any board layout

Every card gets a hard two-state contract. Collapsed is the four-question tile — and is the only resting state. Expanded (chevron, or auto-expand when severity is error/attention) reveals the focus layer: agent sessions with terminal/peek controls, milestone timeline, review history, GitHub checks, autonomous stage track. One card expanded per column; state keyed by feature id so live reconciliation (reconcileKeyedCards) preserves it across polls.

C

Board + docked inspector

phase 2 candidatereuses drawer tabskeyboard-first

Cards shrink to one-line tiles; selecting one docks the existing drawer (Spec / Status / Events / Agents…) as a persistent right panel instead of a modal overlay, so you can walk the board with j/k and watch detail update live. This is the "mission control" pattern — strongest for operators running many autonomous features at once. It composes with A and B (the inspector replaces "expanded" on wide screens) but is a larger interaction change, so I'd sequence it after the core lands.

D

Stage swimlanes (rows, not columns)

radical alternativefull-width active cards

Flip the axis: stages become horizontal lanes. The In-Progress lane renders full-width row-cards — a card gets the entire board width, so fleet agents, autonomous stage tracks, and close-readiness fit on one uncramped line. Inactive stages render as dense chip-rows. This solves the width problem completely and reads like a flight-status board, but it retires the kanban mental model (and column drag-and-drop) — a bigger bet than A+B. Worth prototyping behind a view toggle if the fleet/autonomous mix keeps growing.

Part 4

The design pattern document & reference

F650's dashboard-card-design.md + wireframe was the right instinct but only covers cards. To stop clutter re-accreting, the pattern reference needs to become the vocabulary every dashboard change is composed from:

  • docs/dashboard-design-system.md — tokens (color, type, spacing, radius), the glance/focus/act altitude model, severity/tone rules, badge & chip inventory with admission criteria ("a new badge must replace one or move to focus level"), action-priority rules, motion & density rules.
  • Live component gallery — evolve card-design-wireframe.html into a rendered gallery page (served in dev, e.g. /design) showing every card state built from the real CSS, so visual review of a new state is "add it to the gallery" not "reproduce it in a sandbox". The gallery page becomes the Playwright visual-regression surface.
  • Component helpers — extract js/components/ (chip, badge, headline, agent-chip, milestone-strip, action-row) used by both pipeline.js and monitor.js, killing the duplicated renderers (P5). New sections must go through the vocabulary or they don't render.

Recommendation

Ship A + B + E as the dash-ux set (features 1–6). They fix all five named problems, keep the kanban mental model, and are individually shippable. Hold C (inspector) as phase 2 once the vocabulary exists, and treat D (swimlanes) as a prototype behind a view toggle only if fleet/autonomous density keeps growing.
